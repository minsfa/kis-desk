## 2. 오늘·내일 계획과 실현 방법

| 일자 | 내용 |
|---|---|
| **수 (오늘 밤)** | 해외(미국) 라이브 매수→체결확인→매도. 미국 정규장 ~22:30~05:00 KST |
| **목·금** | 국내+해외, **10만원 미만 상품** 단순 매수매도 (일반주식·ETF·레버리지) |

**실현 방법: 현재는 100% 수동(manual).**
- 사람이 CLI 명령을 직접 실행해 주문을 트리거한다. (예: `python -m src.cli usbuy SCHD 1 33 --live`)
- **주문 실행 로직 자체는 Python 코드**가 담당(인증·안전게이트·로깅). LLM이 주문을 결정하지 않는다.
- **자동매매(스케줄/규칙 자동 실행)는 아직 아님.** PoC로 배관·체결·비용을 검증한 뒤(다음 단계)에 규칙화·자동화로 넘어간다.
- 레버리지/인버스(예: 인버스2X 252670 ~120원)는 기본예탁금 미달로 **거부될 가능성** → "거부 메시지 확인"도 테스트 결과로 본다.

**진행 단계(로드맵):** M1 API기반(현재, 거의 완료) → M2 전략 방향 결정 → M3 (필요시) 도미노 파이프라인 수리 → M4 전략 엔진+백테스트 → M5 실전 소액 운영.

## 3. OpenClaw 연계 상황

**현재: KIS 자동매매와 OpenClaw는 아직 직접 연결되지 않음.** 독립 프로젝트(`~/kis-autotrade`)로 분리 동작.

- OpenClaw(madu_bot 텔레그램 비서)는 별개로 가동 중 — 일일 보고, 도미노 모니터링 등.
- **계획된 연계(미구현):** KIS의 `report` 명령이 당일 주문/체결/손익 요약 텍스트를 생성 → OpenClaw가 로그(`logs/*.jsonl`)를 읽어 madu_bot으로 텔레그램 보고. PoC 1단계 성공 확인 후 연결 예정.
- **역할 분담 원칙:** 주문 실행 = Python 코드. OpenClaw = 관찰·보고만(주문 결정에 LLM을 두지 않음 — 응답지연·일관성 위험 회피).

## 4. OpenClaw 초기 조사 내용 및 로직 평가

**(가) 초기 조사 — "레버리지 ETF 일중 자동매매 계획안"**
- 구성(아키텍처·시나리오 A/B/C·리스크 규칙)은 잘 짜였으나, **프로젝트를 가장 고위험 전략(2배 레버리지 ETF 일중 단타)으로 확정**해버린 점이 문제.

**(나) 심층 검증 리포트 평가 — 결론은 타당**
- 레버리지 일중 단타는 **음(-)의 기대값** 증거가 강함(자본시장연구원 개인 누적손실). 손익분기 승률 ~55% 필요.
- **KIS는 과거 분봉 미제공** → 인트라데이 백테스트 곤란. 모의투자는 단타 검증에 부적합.
- **거시신호 기반 ETF 스윙**이 거래비용·백테스트 가능성에서 구조적으로 유리.
- 남은 확인 필요: 주문 tr_id 공식 확인(라이브로 일부 검증됨), 레버리지 기본예탁금/사전교육.

**(다) 도미노 모니터링 데이터 평가**
- **신호 설계는 A급**(6노드 가중 DRI + 모멘텀 + 13개 트리거). 단 KR 진입신호가 아니라 **글로벌 리스크 레짐 필터**.
- **데이터 자산은 D급**: 누적 원장 사실상 1일치, 누적이 끊김.
- **운영 실태:** 엔진(`~/domino/domino.py` + CSV 원장)이 heartbeat 파이프라인에서 분리(orphaned)됨. 매일 텔레그램에 나가는 리포트는 **LLM이 그때그때 손으로 작성**하는 것이고 엔진/원장으로 흘러들지 않음. 6/1 이후 누적 정지.
- **활용 방향:** 도미노 DRI는 진입신호가 아니라 **리스크 필터/포지션 사이징 오버레이**로. 가치를 쓰려면 ① 누적 파이프라인 수리(경로통일+자동실행) ② 공개데이터로 과거 DRI 역산(백테스트).

**(라) 종합 결론**
- 전략(레버리지 단타 vs 거시 스윙)은 **미정**. 먼저 **KIS API 토대(모의·실전 양쪽)**를 만들고(현재 단계), 그 위에서 데이터로 전략을 결정한다.
- 1차 목표는 수익이 아니라 **시스템 구축·소액 검증 학습**.

## 부록 — API 키 재발급 절차 (필요 시)
1. KIS 개발자포털(apiportal.koreainvestment.com) 로그인 → 마이페이지/계좌관리의 API 신청현황.
2. 해당 앱의 APP Secret **재발급**(재발급 시 기존 키 무효화).
3. 새 APP_KEY/APP_SECRET를 `config/.env`에 교체(채팅창에 붙여넣지 말고 파일 직접 편집).
4. 토큰 캐시 삭제: `data/state/token_prod.json` 제거 후 `python -m src.cli token --env prod`로 재발급 확인.
